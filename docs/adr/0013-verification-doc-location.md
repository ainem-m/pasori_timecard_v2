# ADR 0013: 実機検証・E2E・受け入れ手順の文書配置

- **日付**: 2026-04-22
- **状態**: Accepted
- **関連**: AGENTS.md, ADR 0006, docs/spec/05_audit_and_backup.md, docs/spec/07_security.md

## スコープ

本 ADR は、**手動または半手動で行う実機・実 UI・実外部サービスを通した検証文書の配置**
を定める。

自動テストコード、テストフレームワーク、CI 設定そのものの配置は対象外とする。

## 背景

Terminal の実機打刻、Admin のブラウザ操作、LINE WORKS 実送受信のような確認は、
単体テストや結合テストの延長ではなく、**実機・実 UI・実外部サービスを通す検証**
として扱う必要がある。

一方で、これらの情報をどこに置くかを決めていないと、毎回以下の迷いが発生する。

- `docs/testing/` を新設するべきか
- `docs/spec/` に全部書くべきか
- runbook 的な手順を `docs/archive/` に置いてよいか

この状態だと、後から文脈なしで見返したときに**毎回別の配置判断**になりうる。

## 決定

### 1. 永続的に守るべき検証要求は `docs/spec/` に置く

- 何をリリース前に確認しなければならないか
- どの機能が受け入れ確認対象か
- どの外部連携が実動確認必須か
- 何が観測されれば合格か

これらは製品仕様・運用要件の一部であり、`docs/spec/` に属する。

例:

- Terminal 実機打刻 happy path を受け入れ確認対象にする
- オフライン打刻復旧を受け入れ確認対象にする
- LINE WORKS callback の実送受信確認を受け入れ確認対象にする

特に実機操作を伴う検証要求については、単に「確認すること」ではなく、
**操作またはイベントに対して観測されるべき期待結果と合格基準**を明示する。

### 2. 実施手順・チェックリストは `docs/verification/` に置く

- 手順書
- 手動 E2E チェックリスト
- リリース前確認表
- 実施時の確認ポイント
- 実施環境、使用機材、前提条件、操作順序、確認方法、証跡取得方法

これらは仕様本文ではなく、**検証運用の実務文書**であるため、
`docs/spec/` から分離して `docs/verification/` に置く。

`docs/verification/` の各手順書は、対応する `docs/spec/` の要求を参照する。

### 3. `docs/testing/` は新設しない

理由:

- このリポジトリでは `testing` という語が「テストコード」「テスト戦略」
  「実機受け入れ手順」を混在させやすい
- 今回必要なのはテストフレームワークの説明ではなく、
  **実機・実サービスを使う確認手順の置き場**である
- `verification` の方が対象を限定できる

### 4. 一時的なメモは `docs/archive/` または issue に置き、固まったら昇格する

- 調査メモ
- 未確定のチェック案
- 一時的な叩き台

これらは確定文書ではないため、直接 `docs/spec/` や `docs/verification/` に置かない。
`docs/archive/` 上の文書は**正本として参照しない**。

## 配置ルール

### `docs/spec/`

- 「何を確認しなければならないか」を書く
- 検証対象、必須条件、合格基準を書く
- 実機操作を伴う検証要求については、単に「確認すること」ではなく、
  操作またはイベントに対して観測されるべき期待結果と合格基準を明示する
- 検証項目の要否を決める正本とする

### `docs/verification/`

- 「どう確認するか」を書く
- 実施環境、使用機材、前提条件、操作手順、確認方法、証跡取得方法を書く
- `docs/spec/` の対応要求への参照を持つ
- `docs/spec/` の要求を実施可能にする従属文書とする

### `docs/archive/`

- 現行の正本ではない資料を置く
- 確定前のメモ、過去資料、調査結果を置く
- archive 上の文書は正本として参照しない

## 例

### 良い例

- `docs/spec/05_audit_and_backup.md`
  - オフライン打刻復旧は受け入れ確認対象とする
  - Server 停止中に発生した打刻は、復旧後に失われず同期されなければならない
  - 同期後、打刻は Admin 画面または監査記録から確認できなければならない
  - 同一打刻が重複登録されてはならない
- `docs/verification/e2e-manual-checklist.md`
  - Server を停止する
  - Terminal でテストカードをかざす
  - 一時保存状態を確認する
  - Server を復旧する
  - Admin 画面で同期結果を確認する
  - 監査ログと重複有無を確認する
  - スクリーンショットを保存する

### 悪い例

- `docs/testing/e2e.md`
  - 自動テスト方針と手動受け入れ手順が混ざる
- `docs/archive/e2e-checklist.md`
  - 現役の確認手順なのに archive に置く

## 結論

以後、

- **検証要求**は `docs/spec/`
- **実施手順**は `docs/verification/`
- **一時メモ・過去資料**は `docs/archive/`

に置く。`docs/testing/` は作らない。

なお、実機操作を伴う検証要求については、`docs/spec/` に
「何を操作するか」だけでなく「何が観測されれば合格か」を明示する。
`docs/verification/` は、それを確認するための具体的手順と証跡取得方法を記述する。
