# ADR 0004: TDD 規約採用 (t-wada 流)

- **日付**: 2026-04-16
- **状態**: Accepted
- **関連**: AGENTS.md §6, ADR 0009

## 背景

本プロジェクトはコーディングエージェント (Claude Code / Codex 等) による実装
を前提とする。エージェントは放置すると「実装してからテストを書く」傾向があり、
結果として回帰テストが貧弱になりがち。

一方、本プロダクトは勤怠データという**訂正が重い業務データ**を扱うため、
domain ロジックの正しさを継続的に担保する必要がある。

## 決定

t-wada (和田卓人) 流の TDD を開発規約として採用する。詳細は AGENTS.md §6 を
参照。要点のみここに再掲する。

### 原則

- Red → Green → Refactor を厳守
- 失敗を示すテストなしにプロダクションコードを書かない
- 最小のコードで通す (仮実装 OK)
- Green 中のみリファクタリング

### 手順 (1 機能 = 1 サイクル)

1. TODO リストを書き出す
2. 失敗するテストを書く (Red)
3. 最小のコードで通す (Green)
4. 三角測量で一般化
5. リファクタリング
6. TODO を消す、新 TODO を追加

### テストコードの書き方

- テストの**仕様記述は日本語で表現**
- 識別子ベースのテストは ADR 0009 に従い、英語識別子 + 日本語コメントを使う
- 表示名ベースのテスト (`it(...)` など) は日本語の表示名を使う
- Given-When-Then / Arrange-Act-Assert
- **DRY より DAMP** (可読性優先)

### 適用範囲

| 層 | 方針 |
|---|---|
| `core::domain` | 必須 TDD、proptest 併用 |
| `core::application` | 必須 TDD、use case ごとに integration test |
| `infra_sqlite` | 実 SQLite に対する統合テスト |
| `server` handler | axum_test で request/response |
| `terminal` / `web` | WebDriverIO (E2E) + Vitest |

## Definition of Done への組み込み

以下を同一 PR 内に含めない PR は merge しない:

- 新規コード行に対応するテスト
- テストの仕様記述が日本語で読めること
- 全検証コマンドの green

## 代替案と却下理由

- **実装優先 + 後追いテスト**: エージェントが陥りやすいパターン。回帰防止にならない。
- **テストカバレッジ N% を必須化**: 目標値を設定するとテストの「質」より「量」を追うようになりがち。カバレッジは参考値に留める。

## 備考

- 仮実装 (ハードコードで通す) を許容する。三角測量で一般化するプロセスを尊重する。
- UI E2E は全部 TDD で書くのは過剰なので、happy path のみ必須。
- proptest は domain policy 検証で効果が大きい。打刻推定、締め日計算、勤務時間計算で積極的に使う。
